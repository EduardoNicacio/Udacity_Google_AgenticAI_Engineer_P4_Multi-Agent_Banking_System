# Rubric

Use this project rubric to understand and assess the project criteria.

## Core Agent Implementation

### Criteria

Secure Database Tool Configuration

### Submission Requirements

- The .env file is not submitted.
- Database connection uses environment variables for host, user, and password (not hardcoded credentials).
- Tools are defined in tools.yaml files for both deposit and loan agents.
- Deposit tools include: account listing, balance queries, transaction queries, and a minimum balance check.
- Loan tools include: outstanding balance, loan terms/details, and next payment date.

### Criteria

Agent Tool and Prompt Integration

### Submission Requirements

- Each agent loads its tools and includes them in the agent's tool list.
- Each agent's prompt file describes the available tools and how to use them.
- The deposit agent prompt includes a guardrail instruction to NOT reveal the total balance of all accounts.

### Criteria

Agent Discovery via Agent Cards

### Submission Requirements

- Each agent (deposit, loan, manager) has an agent card JSON file.
- Each agent card includes: name, url, description, and at least one skill.
- The URL reflects where the agent is accessible.

## Multi-Agent Communication

### Criteria

A2A-Based Agent Routing

### Submission Requirements

- The manager agent connects to deposit and loan agents using RemoteA2aAgent.
- The manager agent's prompt provides clear routing instructions for deposit vs. loan questions.
- The manager agent answers basic general banking questions directly (e.g., hours, contact, services) without routing.
- The loan approval workflow uses A2A to communicate with the deposit agent for equity checks.
- Deposit and loan agents do not import each other's code; cross-agent collaboration occurs via A2A only.

### Criteria

Loan Approval Orchestration and State Management

### Submission Requirements

- The deposit agent includes a "minimum balance check" tool that takes a number and returns true/false based on whether total deposits exceed that value. This tool is used by the loan approval workflow.
- The loan approval workflow includes sub-agents for:
- Extracting loan request details (type and amount)
- Getting outstanding loan balance
- Loading and evaluating policy criteria from a PDF in cloud storage
- Computing minimum required equity (using a custom agent, not an LLM for math)
- Checking equity via A2A with the deposit agent
- Loading and evaluating customer profile from a PDF in cloud storage
- Sub-agents use `output_schema` and `output_key` to store results in state.
- Orchestration uses both `SequentialAgent` and `ParallelAgent` to coordinate sub-agents.

### Criteria

Loan Decision Logic and Safe Responses

### Submission Requirements

- A final decision agent evaluates equity check results and customer rating to approve or reject the loan.
- Approved responses mention a loan officer will contact the customer.
- Rejected responses are polite and do NOT reveal specific policy details, thresholds, or customer ratings.

## Testing and Analysis

### Criteria

Functional Testing and Evidence

### Submission Requirements

- Test result files are submitted: `test_results.csv`, `test_results.json`, and `test_results.txt`.
- Screenshots demonstrate the loan approval workflow via the loan agent's web interface, showing:
  - At least one successful loan approval with final state visible
  - At least one loan rejection with final state visible
  - Test results show guardrails working:
    - Requests for total deposit balance are refused or redirected (not revealed)
    - Unauthorized actions (e.g., adding funds) are refused

### Criteria

Report and Risk Analysis

### Submission Requirements

- Report explains the multi-agent architecture, including roles of each agent and how they interact.
- Report provides detailed explanation of the loan approval orchestration.
- Report evaluates test results, identifying at least one strength and one area for improvement.
- Report identifies at least two risks of using agents in banking (e.g., LLM hallucinations, removing humans from decision loops, insufficient guardrails, non-deterministic behavior).
- Report suggests plausible mitigations for each identified risk.
- Report includes at least two suggestions for future system improvements.

## Suggestions to Make Your Project Stand Out

1. Run the agents on different ports or machines.

    The agent cards for the deposit, loan, and manager agents are updated to reflect the new locations. The submission includes evidence that the agents are running on different ports or machines and can still communicate with each other.

2. Save loan applications to a database.

    A new tool is created to save the loan application and the agent's decision to a database. The loan approval sub-agent is updated to use this new tool. The submission includes evidence that the loan applications are being saved to the database.

3. Create a loan manager agent to review applications.

    A new agent is created for loan managers. The agent has tools to review the loan applications saved in the database. The submission includes evidence of the loan manager agent functioning correctly.

4. Expand test coverage.

    Additional tests for other loan types and amounts are added to test_scenarios.csv. The report and analysis include the results of these additional tests. Additional customer profiles are created and used for testing.
